# arXiv AI/ML 논문 자동 수집 에이전트 — 시장조사보고서

> **작성자**: 주혁 | **날짜**: 2025년 | **버전**: v1.0

---

## 🎯 Executive Summary

이 보고서는 arXiv 논문 자동 수집·요약 에이전트의 **시장성, 경쟁 환경, 수익 구조, 리스크**를 분석합니다. 팀 전원의 관점(승우·지민·주혁·수영·유진·민아)을 반영하여, 기술적 실현 가능성과 사용자 경험, 운영 안정성을 균형 있게 검토했습니다.

## 1. 시장 규모 (TAM / SAM / SOM)

|------|------|-----------|

## 2. 경쟁사 분석

### 2.1 경쟁 구도 지도

```
                    [요약 품질 높음]
                          ↑
                    Elicit · Consensus
                          |
[영어 특화] ←————————————+————————————→ [한국어 특화]
                          |
            Semantic Scholar · Connected Papers
                          |
                          ↓
                    [요약 품질 낮음]

→ 우리 포지셔닝: 오른쪽 상단 (한국어 + 고품질 요약)
```

### 2.2 주요 경쟁사 상세 분석

| 경쟁사 | 강점 | 약점 | 우리와의 차별점 |
|--------|------|------|----------------|
| **Semantic Scholar** | 방대한 논문 DB, 무료 API | 한국어 요약 없음, 커스터마이징 불가 | 한국어 요약 + 키워드 커스터마이징 |
| **Elicit** | AI 기반 분석 우수 | 유료($10/월~), arXiv 특화 아님 | arXiv 특화 + 무료 오픈소스 베이스 |
| **Consensus** | 인용 기반 신뢰도 | 영어 전용, 실시간 아님 | 매일 자동 수집 + 한국어 |
| **Connected Papers** | 시각화 탁월 | 요약 없음, 수동 탐색 | 자동화 배치 + 요약 제공 |
| **PaperWithCode** | 코드 연동 | ML 특화, 요약 기능 없음 | 3줄 요약 + 한국어 |
| **arxiv-sanity** | arXiv 특화 | UI 노후화, 요약 없음 | Claude API 기반 고품질 요약 |

> **주혁 관점**: 기존 도구 그대로는 아까워요. "한국어 + arXiv 특화 + 자동 배치"라는 조합이 현재 시장에 없습니다. 이게 진짜 기회입니다.

### 2.3 경쟁 우위 요소

```
우리만의 경쟁 우위 (현재 시장에서 조합이 없는 것):

1. ✅ arXiv 공식 API 특화 (feedparser 기반 안정성)
2. ✅ Claude API 기반 한국어 3줄 요약 (GPT 계열 대비 맥락 이해 우수)
3. ✅ 키워드 커스터마이징 (config.yaml 기반)
4. ✅ 로컬 실행 가능 (데이터 프라이버시 민감 기업 대상)
5. ✅ 오픈소스 베이스 → 진입 장벽 낮음
```

## 3. 타겟 고객 분석

> **민아 관점**: "누가 매일 이걸 쓸 건가?"가 핵심입니다. 기술 스펙이 아니라 사용자 행동 패턴에서 출발해야 합니다.

### 3.1 페르소나 정의

#### 🎯 Primary — 국내 AI 스타트업 연구팀

```
대표 페르소나: 김지훈 (ML 엔지니어, 32세)
- 회사: Series A 단계 AI 스타트업, 연구팀 3명
- 문제: 매주 arXiv 논문 50~100개 나오는데, 다 읽을 시간 없음
- 현재 방법: 트위터·링크드인에서 누군가 공유해줄 때까지 기다림
- 지불 의사: 팀 단위 월 $30~50

Pain Points:
  - 영어 논문 읽는 데 시간 과다 소비
  - 키워드 트래킹 자동화 안 됨
  - "이번 주에 RAG 관련 뭐 나왔지?" 질문에 즉답 불가
```

#### 🎯 Secondary — 대학원생 · 연구자

```
대표 페르소나: 박수진 (AI 박사과정, 27세)
- 소속: 국내 AI 연구실
- 문제: 지도교수에게 "최신 동향 파악하고 있냐?" 질문 받음
- 현재 방법: 매주 수동으로 arXiv 검색, Google Scholar Alert
- 지불 의사: 개인 월 $5~15 (학생 할인 필요)

Pain Points:
  - 수동 모니터링 시간 소비 (주당 2~3시간)
  - 영어 초록을 빠르게 스캔하는 능력 필요
  - 분야별 트렌드 파악 어려움
```

#### 🎯 Tertiary — 기업 내 AI 리서치 인텔리전스 팀

```
대표 페르소나: 이민경 (AI 전략팀, 38세)
- 소속: 대기업 AI Lab 또는 컨설팅펌
- 문제: 임원에게 주간 AI 트렌드 리포트 제출 필요
- 현재 방법: 인턴이 수동으로 수집·번역
- 지불 의사: 팀/기업 월 $200~500

Pain Points:
  - 인력 리소스 낭비 (인턴 시간)
  - 일관된 품질 유지 어려움
  - 자동화 요구 높음
```

## 5. 리스크 분석

> **지민·수영·유진 관점 반영**: 리스크를 "나중에 생기면 처리"가 아니라, **지금 코드 구조에서 어떻게 대응하는지**까지 포함합니다.

### 5.1 기술 리스크

| 리스크 | 발생 가능성 | 영향도 | 대응 전략 |
|--------|------------|--------|----------|
| **Claude API 타임아웃/실패** | 중 | 중 | 재시도 1회 + 원본 초록 폴백 저장 (수영 제안 반영) |
| **arXiv API 구조 변경** | 저 | 고 | feedparser 버전 고정 + 파싱 로직 모듈화 |
| **config.yaml 파싱 실패** | 중 | 고 | `sys.exit(1)` 명시적 실패 처리 (지민 요구사항) |
| **중복 논문 수집** | 중 | 저 | 파일 스캔 기반 중복 체크 (유진 제안) |
| **API 키 노출** | 저 | 매우고 | `.env` 분리 + `.gitignore` 필수 |

```
기술 리스크 대응 코드 구조 (수영·지민·주혁 합의):

# summarizer.py 핵심 구조
def summarize_with_fallback(abstract, title):
    try:
        summary = call_claude_api(abstract)
        logger.info(f"[SUCCESS] {title}")
        return summary
    except Exception as e:
        logger.warning(f"[RETRY] {title}: {e}")
        time.sleep(3)
        try:
            summary = call_claude_api(abstract)
            logger.info(f"[RETRY_SUCCESS] {title}")
            return summary
        except Exception as e2:
            logger.error(f"[FAILED] {title}: {e2}")
            return f"[요약 실패 - 원본 초록]\n{abstract}"
```

### 5.3 운영 리스크

> **민아 관점 반영**: 사용자가 "어제 왜 논문이 없지?"를 알 수 없는 구조가 가장 큰 운영 리스크입니다.

```
운영 리스크 시나리오:

[시나리오 1] 에러 무음 실패
  문제: API 실패 → 로그 없음 → MD 파일에 요약만 없음
  사용자: "어? 오늘 논문 수가 적네" (3주 뒤 발견)
  대응: 민아 요구사항대로 output/ERROR_YYYY-MM-DD.md 생성

[시나리오 2] Config 오작동
  문제: YAML 오류 → 이전 설정으로 폴백 → 사용자 모름
  사용자: "키워드 바꿨는데 왜 예전 논문이 나오지?" (1주 뒤 발견)
  대응: 지민 요구사항대로 sys.exit(1) + 명시적 에러 메시지

[시나리오 3] 데이터 누락 누적
  문제: 요약 실패가 로깅 안 되면, 패턴 파악 불가
  사용자: "RAG 논문이 최근 2주 요약이 없네?" (원인 불명)
  대응: 수영 요구사항대로 ERROR 섹션 + 재시도 로그 필수
```

## 6. 기술 구현 로드맵

> **승우 관점 반영**: "오늘 PR"이 우선. 단, 팀 합의에 따라 에러 처리는 처음부터 포함.

### Phase 1 — MVP (Week 1)

```
arxiv-agent/
├── config.yaml          ← keywords, max_papers, error_handling 정책
├── agent.py             ← feedparser + 중복 체크 + MD 저장
├── summarizer.py        ← Claude API + 재시도 1회 + 원본 폴백
├── output/
│   ├── 2025-01-15.md   ← 일일 수집 결과
│   └── ERROR_2025-01-15.md  ← 에러 발생 시 (민아 요구사항)
└── .gitignore           ← .env, output/ 제외
```

### Phase 2 — 안정화 (Week 2~4)

```
- 대시보드 추가 (최소 HTML 1개, 민아 요구사항)
- 슬랙/이메일 알림 연동
- 재시도 실패 논문 다음날 자동 재수집 (민아 요구사항)
- 통계 리포트: 7일 요약 성공률 추적
```

### Phase 3 — 확장 (Month 2~3)

```
- LLM Provider 추상화 (Claude → GPT-4o 전환 가능, 주혁 제안)
- config.yaml에 error_handling 정책 설정 (수영 요구사항)
- JSON 출력 포맷 옵션 추가
- GitHub Actions 자동화 (매일 오전 9시 실행)
```

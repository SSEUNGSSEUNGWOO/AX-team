# arXiv AI/ML 논문 자동 수집 에이전트 — 요구사항 정의서

---

## ⚠️ 작성 전 검토 메모 (지민)

요구사항 정리 시작 전에 팀 의견 중 **충돌하는 지점**부터 명확히 했습니다. 그냥 다 넣으면 나중에 구현 단계에서 "이게 맞아요?"가 반드시 나오거든요.

| 충돌 지점 | 승우 | 지민/수영/유진/주혁 | 민아 | **채택 방향** |
|---|---|---|---|---|
| Claude API 재시도 | 불필요 | 1회 재시도 필수 | 재수집 메커니즘 | **재시도 1회 + 폴백 저장** (4:1) |
| 에러 처리 방식 | 원본 저장 후 진행 | 명시적 로깅 필수 | ERROR 파일 생성 | **로깅 + MD 파일 내 명시** |
| 대시보드 HTML | 해당 없음 | 해당 없음 | 필요 | **MVP 외 범위로 분리** |
| 재수집 메커니즘 | 불필요 | 로그만 | 자동 재수집 | **로그 기록만 (자동 재수집은 v2)** |

> 민아님 대시보드/자동 재수집 제안은 방향은 맞는데, MVP 범위를 벗어납니다. v2 항목으로 명시하겠습니다.

---

## 1. 프로젝트 개요

### 1.1 목적

arXiv에서 AI/ML 관련 논문을 매일 자동으로 수집하고, Claude API를 이용해 한국어 3줄 요약을 생성하여 마크다운 파일로 저장하는 배치 에이전트를 구축한다.

### 1.2 시스템 컨텍스트

```
config.yaml
    │
    ▼
agent.py (진입점)
    │
    ├─ arxiv API (feedparser) → 논문 메타데이터 수집
    │
    ├─ 중복 체크 (기존 output/ 파일 스캔)
    │
    ├─ summarizer.py → Claude API 호출 (재시도 1회 포함)
    │
    └─ output/YYYY-MM-DD.md 저장
```

### 1.3 프로젝트 구조

```
arxiv-agent/
├── config.yaml          # 키워드, 수집 수, 에러 처리 정책
├── agent.py             # 메인 오케스트레이터
├── summarizer.py        # Claude API 호출 + 재시도 로직
├── output/              # 일일 결과 마크다운 저장
│   └── YYYY-MM-DD.md
├── logs/                # 실행 로그 저장
│   └── YYYY-MM-DD.log
├── requirements.txt
├── .env.example         # API 키 환경변수 예시
└── .gitignore
```

### 1.4 기술 스택

| 항목 | 선택 | 근거 |
|---|---|---|
| Python | 3.10+ | 팀 합의 |
| arXiv API | feedparser | 요구사항 명시 |
| LLM API | anthropic (Claude) | 요구사항 명시 |
| 설정 파일 | PyYAML | config.yaml 파싱 |
| 로깅 | Python logging (표준) | 별도 라이브러리 불필요 |
| 스케줄링 | cron 또는 외부 스케줄러 | 에이전트 자체는 단순 배치 스크립트 |

> **왜 스케줄러를 직접 구현하지 않나요?** 에이전트 내부에 스케줄러까지 넣으면 단일 책임 원칙 위반입니다. cron으로 `python agent.py` 하루 1회 실행하는 게 더 명확합니다.

## 2. 기능 요구사항

### FR-01. 설정 파일 로딩

**우선순위: P0 (필수)**

#### FR-01-1. config.yaml 구조

```yaml
keywords:
  - LLM
  - multi-agent
  - RAG

max_papers: 10          # 하루 최대 수집 논문 수 (기본값: 10)
language: ko            # 요약 언어 (현재 ko 고정)

error_handling:
  claude_api_max_retries: 1       # 재시도 횟수 (수영 요구사항)
  claude_api_retry_delay_sec: 3   # 재시도 대기 시간(초)
  on_summary_failure: "store_original"  # store_original | skip
```

#### FR-01-2. 파싱 실패 처리 (주혁, 지민, 승우 합의)

- `config.yaml` 파일이 없거나 YAML 문법 오류 시: **앱 구동 즉시 실패**
- `sys.exit(1)` + `logger.error()` 로 명확한 에러 메시지 출력
- **기본값 폴백 절대 금지** — 사용자가 실수를 인지해야 함

```python
# 구현 기준 예시
try:
    with open("config.yaml") as f:
        config = yaml.safe_load(f)
except FileNotFoundError:
    logger.error("config.yaml 파일을 찾을 수 없습니다.")
    sys.exit(1)
except yaml.YAMLError as e:
    logger.error(f"config.yaml 문법 오류: {e}")
    sys.exit(1)
```

> **왜 폴백 안 하나요?** 사용자가 키워드를 수정했는데 문법 오류가 있을 때, 이전 설정으로 조용히 실행되면 "어제 왜 다른 결과가 나왔지?" 디버깅이 불가능합니다. 명시적 실패가 훨씬 안전합니다.

### FR-02. 논문 수집

**우선순위: P0 (필수)**

#### FR-02-1. arXiv API 연동

- `feedparser` 라이브러리로 arXiv Atom 피드 파싱
- `config.yaml`의 `keywords` 목록 각각에 대해 arXiv 검색 쿼리 실행
- 키워드별 검색 후 **중복 제거** (동일 arxiv ID 기준)
- `max_papers` 초과 시 최신순으로 자름

#### FR-02-2. 수집 항목

각 논문마다 다음 항목 수집:

| 항목 | arXiv 필드 | 비고 |
|---|---|---|
| 제목 | `entry.title` | 필수 |
| 저자 | `entry.authors` | 쉼표 구분, 최대 5명 + "외 N명" |
| 초록 (원문) | `entry.summary` | 요약 실패 시 폴백용 |
| arXiv 링크 | `entry.link` | 필수 |
| 논문 ID | `entry.id` | 중복 체크용 |
| 제출일 | `entry.published` | 정렬 기준 |

#### FR-02-3. 중복 체크 (유진 방안 채택)

- `output/` 디렉터리의 기존 MD 파일들을 스캔하여 이미 수집된 arxiv ID 목록 추출
- 동일 ID가 존재하면 해당 논문 건너뜀
- **왜 파일 스캔 방식인가?** 별도 DB/캐시 없이 구현 단순함 유지. 논문 수가 하루 10개 수준이므로 성능 문제 없음.

> **엣지 케이스**: `output/` 디렉터리가 없을 경우 → 자동 생성 후 진행

### FR-03. 논문 요약 생성

**우선순위: P0 (필수)**

#### FR-03-1. Claude API 호출

- `anthropic` 라이브러리 사용
- 입력: 논문 원문 초록
- 출력: 한국어 3줄 요약
- API 키는 환경변수 `ANTHROPIC_API_KEY`로 관리 (코드에 하드코딩 절대 금지)

#### FR-03-2. 재시도 및 폴백 전략 (지민, 수영, 유진, 주혁 합의)

```
Claude API 호출 시도
    │
    ├─ 성공 → 요약 텍스트 반환
    │
    └─ 실패 (타임아웃/API 에러)
           │
           ├─ logger.warning("[RETRY] 논문 제목, 오류 내용")
           ├─ 3초 대기
           └─ 재시도 1회
                  │
                  ├─ 성공 → 요약 텍스트 반환
                  └─ 실패
                         ├─ logger.error("[FAILED] 논문 제목, 오류 내용")
                         └─ 원본 초록 텍스트 반환 + 실패 플래그 True
```

- 재시도 횟수/대기 시간은 `config.yaml`의 `error_handling` 섹션에서 설정
- **실패해도 해당 논문 건너뛰지 않음** — 원본 초록으로 저장하여 데이터 갭 방지 (주혁 요구사항)

> **왜 재시도가 "과설계"가 아닌가?** try-except 블록 하나 추가 = 코드 약 10줄. 이 10줄이 없으면 1개월 뒤 "왜 이 날 요약이 없지?"를 디버깅할 방법이 없습니다. 로그가 없으면 데이터도 없습니다.

#### FR-03-3. 요약 프롬프트 기준

```
다음 논문 초록을 한국어로 3줄로 요약해주세요.
각 줄은 핵심 내용 하나씩 담아야 합니다.

초록: {abstract}
```

### FR-04. 결과 저장

**우선순위: P0 (필수)**

#### FR-04-1. 파일 경로

```
output/YYYY-MM-DD.md
```

- 같은 날 재실행 시: 기존 파일에 **추가(append)가 아닌 덮어쓰기**
  - 이유: 중복 체크가 이미 존재하므로 재실행 시 새로운 논문만 수집됨. 덮어쓰기가 파일 상태를 예측 가능하게 유지함.

> **이 결정에 대해**: 재실행 시 덮어쓰기 vs 추가(append) 중 어느 게 맞는지 팀 확인 필요합니다. 요구사항에 명시가 없어서 덮어쓰기로 일단 정했는데, "같은 날 두 번 실행하면?"이라는 케이스가 있기 때문입니다.

#### FR-04-2. 마크다운 포맷

```markdown
# arXiv 논문 수집 — 2025-01-15

> 키워드: LLM, multi-agent, RAG | 수집: 10편 | 요약 성공: 9편 | 요약 실패: 1편

---

## 1. [논문 제목](https://arxiv.org/abs/XXXX.XXXXX)

- **저자**: 홍길동, Jane Smith, 외 3명
- **제출일**: 2025-01-14
- **링크**: https://arxiv.org/abs/XXXX.XXXXX

### 요약 (한국어)
1. 첫 번째 핵심 내용
2. 두 번째 핵심 내용
3. 세 번째 핵심 내용

---

## 2. [논문 제목](링크)

...

---

## ⚠️ 요약 실패 목록

| 논문 제목 | 링크 | 실패 사유 |
|---|---|---|
| [제목](링크) | [arxiv](링크) | Claude API 타임아웃 (재시도 후 실패) |

> 위 논문들은 원본 초록이 output 파일 내 해당 섹션에 저장되어 있습니다.
```

> **왜 요약 실패 목록을 별도 섹션으로?** (민아 요구사항 반영) 사용자가 파일을 열었을 때 "이 논문은 왜 요약이 없지?"를 즉시 알 수 있어야 합니다. 숨겨두면 안 됩니다.

#### FR-05-2. 로그 레벨 기준

| 레벨 | 상황 | 예시 |
|---|---|---|

| `ERROR` | arXiv API 연결 실패 | `arXiv API 응답 없음: 오류 내용` |

## 3. 비기능 요구사항

### NFR-01. 신뢰성

- **데이터 갭 없음**: 어떤 논문도 "조용히 사라지지 않음" — 요약 실패 시 원본 초록으로 저장
- **실행 결과 추적 가능**: 모든 실패는 로그 + MD 파일 내 실패 목록에 기록
- **환경 독립성**: API 키는 환경변수로만 관리, 코드/설정 파일에 하드코딩 금지

### NFR-02. 유지보수성

- **에러 처리 정책 config화**: 재시도 횟수/대기 시간을 코드가 아닌 `config.yaml`에서 변경 가능 (수영 요구사항)
- **단일 책임**: `agent.py`(오케스트레이션), `summarizer.py`(요약 생성) 분리
- **스케줄링 외부화**: cron 등 외부 스케줄러 사용, 에이전트 자체는 1회 실행 스크립트

### NFR-03. 성능

- 하루 최대 수집 논문 10개 기준 — 성능 최적화 불필요
- arXiv API 연속 호출 시 **1초 간격** 딜레이 (API 남용 방지)

### NFR-04. 보안

- `ANTHROPIC_API_KEY`는 `.env` 파일 또는 환경변수로 관리
- `.env`, `logs/` 는 `.gitignore`에 포함

## 6. 미확정 사항 — 팀 확인 필요

다음 항목은 요구사항에 명시가 없거나 팀 의견이 엇갈려서 **결정 전 확인**이 필요합니다:

| # | 질문 | 현재 임시 결정 | 확인 필요 이유 |
|---|---|---|---|
| Q1 | 같은 날 agent.py를 두 번 실행하면 output 파일 덮어씀? 아니면 추가(append)? | 덮어쓰기 | 명세 없음. 운영 방식에 따라 다름 |
| Q2 | 키워드 여러 개일 때 각 키워드별로 `max_papers`를 적용? 아니면 전체 합산? | 전체 합산 | "LLM 10개 + RAG 10개 = 20개" vs "합쳐서 10개"가 다름 |
| Q3 | arXiv API 실패(네트워크 오류) 시 앱 종료? 아니면 다음 키워드로 진행? | 앱 종료 + 로그 | 논문 수집 자체가 목적이므로 실패 시 빈 파일보다 명시적 실패가 나음 |
| Q4 | 저자 표시 최대 몇 명? | 5명 + "외 N명" | arXiv 논문은 저자 수십 명인 경우 있음 |
| Q5 | `on_summary_failure: skip` 선택 시 해당 논문 MD에 아예 안 넣음? 아니면 제목/링크는 포함? | 제목/링크는 포함, 요약만 생략 | 사용자 입장에서 논문 존재 자체는 알아야 함 |

---

이 정도 정리가 되어야 코드 작성 시작할 수 있습니다. **Q1~Q5 먼저 답변 주시면 바로 구현 들어갑니다.**

---

## 1. MVP 범위 (3단계 파이프라인)

```
[수집] → [정제] → [출력]
  ↓          ↓         ↓
외부API    기준문서   리포트JSON
상권DB     적용 후    PDF렌더링
출점이력   저장
```

### MVP 포함
| 항목 | 내용 |
|------|------|
| 수집 | 공공 상권 API, 출점 이력 CSV 업로드 |
| 정제 | **결측치 임계값 · 이상치 규칙 문서 기반** 자동 필터 |
| 출력 | 입지 점수 + 출점 권고 리포트 (JSON → PDF) |
| 인증 | B2B 기업 계정 (JWT) |
| UI | 리포트 조회 · 다운로드 단일 화면 |

### MVP 제외 (v2 이후)
- RAG 기반 자연어 질의
- 실시간 유동인구 스트리밍
- 경쟁 브랜드 크롤링
- 대시보드 고도화

## 2. 마일스톤

```
Week 0~1   ████  [설계 · 문서화]
Week 2~3   ████  [수집 모듈]
Week 4~5   ████  [정제 모듈]
Week 6~7   ████  [출력 모듈]
Week 8     ████  [통합 테스트]
Week 9~10  ████  [베타 · 피드백]
Week 11~   ████  [v2 착수]
```

### M0 — 설계 · 문서화 완료 `Week 0~1` 🔴 **코딩 전 필수**

| 산출물 | 담당 | 완료 기준 |
|--------|------|-----------|
| 데이터 정제 기준서 (결측치 임계값 · 이상치 규칙) | 수영 | PR 머지 |
| DB 스키마 v1 확정 | 수영 | ERD 리뷰 완료 |
| API 설계서 (OpenAPI 3.0) | 수영 | Swagger 배포 |
| 모듈 경계 정의서 | 전체 | 팀 사인오프 |

> ⚠️ M0 완료 전 구현 코드 머지 **불가** — PR 규칙으로 강제

---

### M1 — 수집 모듈 `Week 2~3`

| 항목 | 내용 |
|------|------|
| 공공 상권 API 연동 | 소상공인진흥공단, 행안부 주소 API |
| CSV 업로드 파이프라인 | 출점 이력 · 심사 기준 파일 |
| 원시 데이터 적재 | `raw_data` 테이블 분리 저장 |
| 수집 실패 로깅 | Dead Letter Queue 구조 |

```
확장성 주의:
raw / processed / output 테이블 분리.
나중에 스트리밍 붙일 때 raw만 교체하면 되도록.
```

---

### M2 — 정제 모듈 `Week 4~5`

| 항목 | 기준 |
|------|------|
| 결측치 처리 | 임계값 초과 시 레코드 제외 + 로그 기록 |
| 이상치 탐지 | IQR · Z-score, 규칙은 **M0 기준서 그대로** |
| 정제 이력 추적 | `quality_log` 테이블 — 왜 버렸는지 기록 |
| 재처리 가능 구조 | 기준 변경 시 raw에서 재실행 가능 |

> 정제 기준은 **코드 하드코딩 금지** → DB/YAML 설정으로 분리

---

### M3 — 출력 모듈 `Week 6~7`

| 항목 | 내용 |
|------|------|
| 입지 점수 산출 | 가중치 모델 (v1 룰베이스) |
| 권고 리포트 생성 | JSON 구조화 → PDF 렌더링 |
| API 엔드포인트 | `POST /analysis` `GET /report/:id` |
| 리포트 버저닝 | 동일 입지 재분석 시 이전 버전 보존 |

---

### M4 — 통합 테스트 `Week 8`

- 파이프라인 E2E 테스트 (수집 → 정제 → 출력)
- 데이터 정제 기준서 ↔ 실제 코드 일치 검증
- 부하 테스트 (동시 리포트 생성 10건)
- 보안 점검 (B2B 계정 격리, 리포트 접근 권한)

---

### M5 — 베타 `Week 9~10`

- 파트너 프랜차이즈 1~2개사 클로즈드 베타
- 피드백 수집 → v2 백로그 반영

---

## 3. v2 범위 `Week 11~`

| 기능 | 전제 조건 |
|------|-----------|
| RAG 자연어 질의 | 정제 파이프라인 안정화 후 |
| 실시간 유동인구 | 스트리밍 수집 모듈 교체 |
| 경쟁사 크롤링 | 법적 검토 완료 후 |
| 멀티 브랜드 대시보드 | 스키마 확장 (brand_id 컬럼 준비는 MVP에) |

> `brand_id` 같은 확장 컬럼은 **MVP DB에 미리 설계**. 나중에 ALTER TABLE 안 해도 되게.
